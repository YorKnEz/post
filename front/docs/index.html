Introduction

Purpose

Realize a Web application for translating poems commented by multiple users. Translation for the poem or a fragment of it may be provided with the afferent annotation. The system will support lyric annotation with the possibility of sharing on social platforms like Tumblr or WordPress, based on the public APIs. Statistics will be provided regarding the activity on the platform (e.g. the most active translator, the most annotated poem, ...). The new translations will be exposed on an RSS flux. The essential functionalities will be provided by our own API, built based on the architectural styles of REST or GraphQL.

Intended Audience and Reading Suggestions

This document is intended for those that want to take a look at our source code. This document provides detailed documentation for our product and is required to read before taking a look at the source code. The document is intended to be read in the given order.

Product Scope

Product that aims to provide internationalization of poems by the community and ease of sharing knowledge about said poems.

  • Approve/deny new lyrics/annotations
  • Approve/deny new edits of lyrics/annotations

User Classes and Characteristics

There will be three types of users which were already sketched in the Product functions section:

  • Regular
  • Poet
  • Admin

Regular

The regular user is the average user that uses the application. He can contribute to lyrics by editing them and annotationg them, but he requires the approval of an admin.

Poet

The poet is a special kind of user that has the ability to do whatever he wants without requesting permission to any piece of content that is his own.

Admin

The admin is the one that approves the requests of the regular users and, in general, handles moderation of the application. He can approve poets on the app, new poems, new annotations, edits, etc.

Operating Environment

The app has been extensively tested only on Firefox browser. It may be used on other browsers as well but we do not guarantee 100% compatibility.

Design and Implementation Constraints

The main limitation of this product is the fact that we are not allowed to use any frameworks for this project, everything must be done from scratch. At the same time, there is no guarantee that the pages of the app are 100% compatbile with other browsers than Firefox.

User Documentation

We do not provide any kind of user documentation. Our application is designed to be easy to use and intuitive and, as such, we do not see the need for a user manual or any other document of that type.

Assumptions and Dependencies

We will use APIs of social media apps in order to provide the share feature.

System Features

In what follows, we will present features by pages (or components if needed)

Navigation

Description and Priority

Priority: HIGH

Since our application includes multiple pages, the need of a navigation tool is mandatory. As such, our app includes a navbar and a sidebar which contain a few useful links in order to make navigation easier. At the same time, they include a few buttons such as a theme changer and a link to our RSS flux. The sidebar contains the useful links and will include the search and icons from the navbar on lower screen widths. The navbar is for quicker search and access to some functionalities. The full list of features included in the navbar are:

  • home page link
  • search bar
  • add poem page link
  • rss feed link
  • theme toggler button
  • profile page link
  • log in link (if not logged)
  • register link (if not logged)
  • log out button (if logged)

Stimulus/Response Sequences

The user will want to navigate to other pages. In order to do that, they must use the navbar/sidebar.

Functional Requirements

Links

When the user clicks a link, they will navigate to that page.

Search

The user types something in the search bar and then he will receive a list of results based on the search, or a page with an error message if the search did not match anything.

Theme

User clicks theme toggler and theme changes from light to dark or vice-versa.

Register

Description and Priority

Priority: MEDIUM

The register page allows users to create accounts on our application. Accounts are necessary for users that want to bring new content to the site, such as: new lyrics, new translations, new annotations etc. Registering is not necesasry to just view the content of the site and, as such, this feature is not of high importance.

Stimulus/Response Sequences

Accounts are necessary for users that want to bring new content to the site, such as: new lyrics, new translations, new annotations etc.

Functional Requirements

Inputs

In order to create an account users must give their first and last name, email, pick a nickname and pick a password.

The nickname must be unique, otherwise an error is given.

The email must be valid, otherwise an error is given.

The passwords must match, otherwise an error is given.

Email verification

The user will receive an email which they have to verify if they want their account to work.

Log In

Description and Priority

Priority: MEDIUM

The login page allows a user to enter their account in order to benefit of the user related features.

Stimulus/Response Sequences

An user that created an account must log in to their account if they want to use it.

Functional Requirements

Inputs

In order to log in, a user must provide their nickname and password.

If the nickname or password don't match, an error is given.

Reset password

Description and Priority

Priority: LOW

If the user forgets their password, they can access this page in order to reset their password.

Stimulus/Response Sequences

The user forgets their password. They come to this page, fill out the email and new password. They get a confirmation mail, after which the password is updated.

Functional Requirements

Inputs

The user must provide their email and new password.

The email must be valid, otherwise an error is given.

The passwords must match, otherwise an error is given.

Home

Description and Priority

Priority: HIGH

The home page presents the first thing a new user sees when they use our app. This page includes:

  • Section of trending poems/translations
  • Section of the most popular poems of all time (or recently, TBD)
  • Section of the most recent poems/translations
  • Section of the most active users of our application

Stimulus/Response Sequences

The user must simply open the application and they will see this main page with the specified content.

Functional Requirements

Consuming content

The user can simply scroll down the page and click the cards shown on each section to learn more about the specific thing they are interested in (e.g. user may click on the card of a top contributor to see their profile and learn more about their contributions).

Profile

Description and Priority

Priority: MEDIUM

The profile page of the user. Here the user can see their name, role and email and their personal statistics, such as number of translations and annotations. Besides, the user may see the poets to which they contributed the most and the history of their activity: annotations, translations and the annotations and translations that are still pending. Lastly, the user has a "Become a poet" button which they might want to press if they want to become a poet.

Stimulus/Response Sequences

The user wants to see their personal statistics or their history. They can access the profile page through the navbar/sidebar.

Functional Requirements

Stats

The user can see their number of annotations and translations

History

The user can see their past annotations and translations and the ones that are still pending by selecting which of those categories to view.

Becoming a poet

If the user wants to become a poet they will press this button and will receive an email with further details regarding the transition to a poet account.

Add poem

Description and Priority

Priority: HIGH

The add poem page is the place where registered users can add new poems or translations for poems.

Stimulus/Response Sequences

If a user wants to add a new poem or translate an already existing one, they must go to this page.

Functional Requirements

Inputs

In order to add a new poem/translation the user must give the following details: author, title, language, lyrics, date, album.

For the author and album inputs, autocompletion is provided based on the existing data in the backend.

The album field is optional.

Poem

Description and Priority

Priority: HIGH

This page allows users to view the lyrics of a poem, the annotations and the translations of it.

Stimulus/Response Sequences

Any user of our app that wants to view the content of poems must access this page.

Functional Requirements

About

The first thing the user sees is the header of the page which provides a bit of insight into the poem's message. The user may click the read more button to navigate to the about section and find even more information about the poem, with the possibility to edit the said poem.

Languages

There is a dropdown called "Languages" right before the lyrics start which provides the user with the ability to change the language in which the lyrics and annotations are displayed.

Viewing annotations

User may click on highlighted text to see additional info (annotations) about said text.

Edit annotations/lyrics

User may click on the edit buttons in order to edit the content of the annotations or lyrics.

Annotating lyrics

User may select text that is not highlighted, a button with the text "Annotate" will appear. Upon clicking it, the text becomes highlighted and a card appears in which the user should input their commentary.

Other features

Dashboard

Description and Priority

Priority: HIGH

The application needs moderation. As such we provide a dashboard page for our admins. This page includes:

  • global statistics that are not avaliable to regular users
  • a feed of poetic requests (i.e. users that want to become poets)
  • a feed of approvals (i.e. requests to add/edit poems/translations/annotations)
  • a feed of users
  • a feed of poems
  • a feed of annotations

Stimulus/Response Sequences

The admins want to moderate the site, they must use this page in order to do so.

Functional Requirements

Statistics

The statistics section provides a list of statistics that are useful in determining the state of the application. These statistics are not visible to other users.

Poetic requests

Regular users may want to be marked as poets in order to be favored by our algorithms and other features. An admin must review such a request and approve/deny it in the poetic requests feed.

Approvals

Regular users may want to add/edit poems/translations/annotations. After they do so, an admin must approve/deny their changes from the approvals feed.

Users, poems, annotations

Feeds for managing the users, poems, annotations with the possibility to delete items from them.
